# Workseed 自动工作

将当前工作区作为已经配置好的开发环境，直接在当前分支工作。支持两种互斥模式：

- **单事种模式**：提示词包含 `事种ID:` 或 `事种ID：`。只处理该 ID，结束后立即退出。
- **队列模式**：提示词不含事种 ID 标记，但明确提供正整数 `projectId` 并要求持续处理队列。

单事种模式优先于队列模式。整个运行只允许一个模式，确定后不得切换。

## 运行约束

- 不要克隆项目、创建工作区、安装整套开发环境或切换 Git 分支。
- 不要执行 `git checkout`、`git switch`、`git reset --hard` 或任何会覆盖用户已有工作的操作。仅在本 Skill 规定的失败回滚流程中，允许对已确认由当前事种修改的明确路径执行精确的 `git restore`。
- 每条事种提交并标记为已完成后，都要及时尝试将当前分支推送到已有上游。推送结果不影响事种完成状态；不要强制推送、推送标签、推送其他分支、修改远端配置或设置新的上游。
- 不要部署、发送消息或执行其他外部副作用，除非用户另行明确授权。
- 允许多个 Agent 共享全局 Workseed MCP；不要维护服务端“当前项目”。单事种模式只使用提示词中的事种 ID，项目 ID 必须从该事种的精确查询结果获得；队列模式只使用提示词中的项目 ID。

## 初始化运行

1. 首先确认当前目录是目标 Git 仓库并检查工作区状态。只要存在任何已暂存、未暂存或未跟踪的修改，立即告知用户“当前存在未提交的修改，不符合启动 workseed-auto-work 的条件”，然后直接退出。不要询问是否继续，不要暂存、提交、回滚或隐藏这些修改，也不要调用任何 Workseed 接口或输出空概览表。
2. 确定运行模式并提取标识：
   - 提示词只要包含 `事种ID:` 或 `事种ID：`，就进入单事种模式。提取标记后第一个明确的正整数作为 `SEED_ID`；即使同时出现 `projectId`，也不得改为队列模式。标记存在但没有有效正整数时询问用户，不要调用 Workseed。
   - 否则进入队列模式，并从提示词提取明确的正整数 Workseed `PROJECT_ID`。未提供时询问用户，不要查询全局队列。
3. 确认已配置 Workseed MCP 工具 `list_seeds`、`get_seed`、`start_seed`、`complete_seed` 和 `skip_seed`。如果工具不可用，说明需要升级或配置 Workseed MCP 服务并停止。
4. 记录当前分支、`HEAD` 和已有上游。不要切换分支；没有上游也不要阻止运行，不要猜测远端、修改远端配置或自行设置上游。
5. 初始化已完成事种、已跳过事种、本次运行已跳过的事种 ID、未完成事种以及运行开始时间。记录每条事种从选中到结束的实际耗时和推送尝试结果。领取令牌只保存在当前运行内存中，不要写入文件、日志、提交信息或退出概览。

## 单事种模式：定位目标

1. 不要调用 `list_seeds`。使用提示词中的 `SEED_ID` 调用：

   ```json
   { "seedId": SEED_ID }
   ```

2. `get_seed` 查询失败时根据错误诊断，包含首次调用最多尝试 3 次；仍无法精确查询则停止。确认返回的事种 ID 等于 `SEED_ID`，`projectId` 是正整数，并将其固定为本次运行的 `PROJECT_ID`。
3. 只允许处理状态为 `inbox` 的目标。目标为 `doing`、`paused`、`skipped`、`done` 或其他状态时直接停止，不要领取、跳过、修改代码或处理其他事种。
4. 如果提示词同时包含 `标题：` 或 `标题:`、`描述：` 或 `描述:`，分别与查询结果的 `title`、`content` 核对。忽略标签后的首尾空白；内容明显不一致时停止并报告，避免处理错误事种。
5. 结合返回的 `projectName`、当前仓库说明和用户提示确认工作区属于该 Workseed 项目。发现明显不匹配时停止，不要调用 `skip_seed` 或修改任何事种状态。
6. 完成领取前检查后，只对该目标执行后续流程。无论最终完成、跳过、被其他 Agent 领取或发生阻塞，都输出概览并退出；不得查询或处理任何其他事种。

## 队列模式：获取下一条事种

1. 严格按以下范围调用 `list_seeds`：

   ```json
   { "projectId": PROJECT_ID, "status": "inbox", "limit": 100 }
   ```

2. `list_seeds` 失败时根据错误进行诊断，最多重试 3 次；仍然失败则停止，不要无限循环。
3. 验证每条返回结果的 `projectId` 都与指定项目一致。发现不一致时停止，并报告服务端项目范围错误。
4. 首次成功获取列表后，结合返回的 `projectName`、当前仓库说明和用户提示确认工作区属于该 Workseed 项目。发现工作区或整个运行环境明显不匹配时停止，不要调用 `skip_seed` 或修改任何事种状态。
5. 只处理 `status` 为 `inbox` 的结果。按接口返回顺序选择第一条不在本次跳过 ID 集合中的事种。服务端已经按照高、中、低优先级排序，不要重新排列。
6. 如果没有未处理的返回结果，说明本项目当前没有可处理的待实现事种，结束运行并输出概览。

## 领取前检查与跳过

领取前阅读事种标题和内容，并检查当前环境。出现以下任一明显缺失条件时，不要调用 `start_seed`：

- 缺少必要凭据、依赖服务、外部资源或用户授权；
- 需求存在关键歧义或缺少必须的产品决策；
- 既无法编写有意义的必要测试，也无法执行有效的静态检查或构建验证。

只将当前事种自身缺少条件视为跳过理由。仓库不匹配、MCP 故障或影响整个队列的环境问题属于运行级阻塞，应停止而不是批量跳过事种。

使用以下参数调用 `skip_seed`，通过 `expectedStatus` 保证只跳过仍处于待实现状态的事种：

```json
{ "seedId": SEED_ID, "expectedStatus": "inbox" }
```

确认返回相同事种 ID、指定项目 ID 和 `skipped` 状态。调用失败或结果不确定时使用 `get_seed` 查询准确 ID：状态为 `skipped` 即视为成功；状态仍为 `inbox` 时包含首次调用最多尝试 `skip_seed` 3 次；状态为 `doing`、`done`、`paused` 或其他值时不要再跳过，单事种模式记录为未完成并退出，队列模式重新获取队列。仍无法确认时停止运行并报告。成功后记录标题、优先级、跳过原因和检查耗时。单事种模式立即输出概览并退出；队列模式将事种 ID 加入本次跳过集合并继续获取下一条。

## 领取并实施

1. 再次确认工作区干净，记录该事种开始时的 `HEAD` 作为 `SEED_BASELINE_HEAD`。工作区出现修改时不要领取事种，立即停止并报告。
2. 为当前事种生成一个不可预测且本次运行唯一的 `CLAIM_TOKEN`，推荐使用随机 UUID。一个事种的所有后续状态操作都必须复用该令牌；不要跨事种复用。
3. 使用以下参数调用 `start_seed`：

   ```json
   { "seedId": SEED_ID, "claimToken": "CLAIM_TOKEN" }
   ```

4. `start_seed` 明确成功，并返回相同事种 ID、指定项目 ID 和 `doing` 状态后开始修改代码。调用失败或结果不确定时使用同一令牌调用 `get_seed`：

   ```json
   { "seedId": SEED_ID, "claimToken": "CLAIM_TOKEN" }
   ```

   根据 `get_seed` 返回的 `seed` 和 `claimedByCaller` 处理：
   - 状态为 `doing` 且 `claimedByCaller` 为 `true` 时，确认当前 Agent 已经领取，直接开始实施；
   - 状态仍为 `inbox` 时，使用同一令牌且包含首次调用最多尝试 `start_seed` 3 次；仍失败则停止运行；
   - 状态为 `doing` 但 `claimedByCaller` 为 `false` 时不要实施；单事种模式立即退出，队列模式重新获取队列；
   - 状态为其他值时不要实施；单事种模式立即退出，队列模式重新获取队列。
5. 实施过程中维护当前事种修改和新增的明确路径列表。工作区意外出现无法归属于当前事种的修改时立即停止，不要继续实施或尝试回滚。
6. 阅读仓库说明和相关代码，将事种标题与内容作为需求，在正常安全和授权范围内完成最小且完整的实现。
7. 必须为实现新增或更新必要的自动化测试。仅修改文档等确实不适用自动化测试的事种，至少运行与改动对应的静态检查或构建验证，并在记录中说明原因。

## 开发、验证与提交：最多十次

将一次“实现或修正、运行必要验证、尝试提交”定义为一次完整尝试。为每条事种将 `ATTEMPT_COUNT` 初始化为 0，最多执行 10 次：

1. 每轮开始时递增 `ATTEMPT_COUNT`，根据上一次的明确诊断实现或修正代码。
2. 运行与本事种直接相关的测试，再运行风险相称的回归检查。测试失败或功能未达到需求时记录原因；未达到 10 次则进入下一轮。
3. 验证通过后检查差异，只暂存本事种产生的文件或代码块。检查暂存区，确认没有无关修改、敏感信息或生成垃圾。
4. 使用以下格式创建 Git 提交：

   ```text
   <prefix>: <事种 ID> - <提炼后的中文标题概述>
   ```

   根据实际改动选择语义化 `prefix`，例如 `feat`、`fix`、`refactor`、`perf`、`docs`、`test`、`build`、`ci` 或 `chore`。标题概述必须准确表达最终实现的核心价值，使用简洁中文重新概括和优化，不要照抄事种标题。例如：

   ```text
   feat: 123 - 支持跨项目筛选事种
   fix: 124 - 修复事种状态统计偏差
   ```

5. 提交命令失败时检查 `HEAD`：如果已经产生包含预期变更的提交，则视为提交成功，避免重复提交；否则记录提交钩子、格式或 Git 错误。未达到 10 次时修正问题并进入下一轮，且必须重新运行验证。不要绕过必要检查。
6. 只有提交成功且 `git status --porcelain` 为空时才结束尝试循环。提交后出现无法解释的工作区修改时停止运行，不要标记事种完成。
7. 不要机械重复完全相同且无望成功的命令；每次重试都必须基于新的诊断或修正。

第 10 次仍未成功提交时执行失败回滚：

1. 确认当前 `HEAD` 仍等于 `SEED_BASELINE_HEAD`，并确认所有待回滚路径均由当前事种产生。任一条件无法确认时立即停止整个运行并报告，不要执行回滚。
2. 对当前事种修改过的已跟踪路径执行精确恢复。禁止使用 `.`、目录根路径、通配符或不受控变量扩大范围：

   ```bash
   git restore --source="$SEED_BASELINE_HEAD" --staged --worktree -- <明确路径列表>
   ```

3. 对当前事种创建的未跟踪文件核对路径后逐个删除，优先使用补丁删除文件。不要删除任何无法确认由当前事种创建的文件，不要递归删除宽泛目录。
4. 使用 `git status --porcelain` 验证工作区恢复干净。仍有修改或出现不一致时立即停止整个运行，不要继续处理后续事种。
5. 不要调用 `complete_seed`。使用领取时的令牌调用 `skip_seed`：

   ```json
   { "seedId": SEED_ID, "expectedStatus": "doing", "claimToken": "CLAIM_TOKEN" }
   ```

6. `skip_seed` 失败或结果不确定时，使用同一令牌调用 `get_seed`：状态为 `skipped` 且 `claimedByCaller` 为 `true` 即视为成功；状态为 `doing` 且 `claimedByCaller` 为 `true` 时，包含首次调用最多尝试 `skip_seed` 3 次；令牌不匹配或状态为其他值时停止运行，不要操作其他 Agent 的事种。
7. 确认事种为 `skipped` 后，记录累计耗时和 10 次尝试的最终失败原因。单事种模式立即输出概览并退出；队列模式继续下一条。

## 完成并尝试推送

成功提交后：

1. 立即使用领取时的令牌调用 `complete_seed`：

   ```json
   { "seedId": SEED_ID, "claimToken": "CLAIM_TOKEN" }
   ```

   确认响应包含相同事种 ID、指定项目 ID 和 `done` 状态。
2. 调用失败或结果不确定时，使用同一令牌调用 `get_seed`：
   - 状态为 `done` 且 `claimedByCaller` 为 `true` 时视为完成成功；
   - 状态仍为 `doing` 且 `claimedByCaller` 为 `true` 时，包含首次调用最多尝试 `complete_seed` 3 次，并在每次失败后重新调用 `get_seed`；
   - `claimedByCaller` 为 `false`、状态为其他值，或者 3 次后仍无法完成时停止运行并报告，不要回滚已经验证并提交的代码。
3. 确认事种为 `done` 后处理推送：
   - 当前分支有上游时，立即执行一次普通 `git push`；禁止使用 `--force`、`--force-with-lease`、`--all`、`--tags`、重写历史或推送其他引用；
   - 没有上游时不要执行 `git push`，记录“未尝试：没有上游”。
4. 无论推送成功还是失败，都只记录本次推送尝试结果。不要回滚提交、改变事种完成状态或反复重试。后续推送可能包含之前尚未推送的提交，因此不要把“本次尝试失败”表述为提交最终未到达远端。
5. 记录事种标题、优先级、提交哈希、本次推送尝试和耗时。优先采用 `complete_seed` 返回的 `durationSeconds`，缺失时使用本地实际耗时。
6. 单事种模式立即输出概览并退出，不调用 `list_seeds`；队列模式返回“获取下一条事种”，直到待实现队列为空。

## 状态完整性

- 队列模式中，当工具支持 `projectId` 参数时始终传入提示词中的项目 ID。单事种模式不得调用 `list_seeds`，项目 ID 只能采用首次精确查询目标事种得到的值。
- 每次 `get_seed` 都必须验证返回 `seed` 的事种 ID 和项目 ID。确认已领取事种时必须传入当前事种的 `CLAIM_TOKEN` 并检查 `claimedByCaller`。查询失败时包含首次调用最多尝试 3 次；仍无法获得准确状态则停止。不要用可能被 `limit` 截断的列表结果代替精确查询。
- 不要使用从未限定项目、其他项目或过期列表结果中获得的 ID。
- 两种模式都只能领取和处理 `inbox` 事种；不要自动领取 `paused`、`skipped`、`doing` 或 `done` 事种。单事种模式不得用队列结果替代提示词指定的 ID，也不得在目标不可处理时选择其他 ID。
- 只对当前运行从 `inbox` 领取且 `claimedByCaller` 为 `true` 的事种继续处理 `doing`、完成或失败跳过流程。不要读取、猜测、输出或冒用其他 Agent 的领取令牌。
- 不要将部分完成、执行失败、未经测试、未提交或被中断的工作标记为已完成。推送成功不是标记完成的前置条件。
- 如果用户中断或替换任务，停止本地工作，不要擅自完成、跳过或自动恢复当前事种。报告其 ID、当前状态和本地工作区状态，但不要输出 `CLAIM_TOKEN`；后续重新启动仍必须遵守“工作区不干净就直接退出”的规则。
- 不要接管任何不属于当前 `CLAIM_TOKEN` 的 `doing` 事种。

## 退出概览

在单事种模式首次成功精确获取目标之前，或队列模式首次成功获取事种列表之前发生的初始化失败，只输出明确原因并直接退出，不输出空表。目标或队列处理开始后，正常退出、中断或因无法恢复的阻塞退出时，以 Markdown 表格汇总本次运行。至少包含：

| 结果 | 事种 ID | 优先级 | 事种 | 耗时 | 提交 | 本次推送尝试或原因 |
| --- | ---: | --- | --- | --- | --- | --- |
| 已完成 | 123 | 高 | 示例功能 | 8分20秒 | `abc1234` | 成功 |
| 已完成 | 124 | 中 | 示例修复 | 3分10秒 | `def5678` | 未尝试：没有上游 |
| 已跳过 | 125 | 中 | 示例任务 | 1分10秒 | — | 缺少必要凭据 |
| 未完成 | 126 | 高 | 示例异常 | 2分30秒 | — | MCP 状态无法确认 |

在表格后给出已完成数量、已跳过数量、未完成数量和总耗时。所有“已跳过”记录都必须已经通过 `skip_seed` 写入 Workseed 的 `skipped` 状态；无法确认状态的事种只能记录为“未完成”。单事种模式的概览只能包含提示词指定的事种；队列为空仅表示当前没有 `inbox` 事种，不代表 `doing`、`paused`、`skipped` 或 `done` 队列为空。
